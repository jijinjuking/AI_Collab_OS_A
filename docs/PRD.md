# AI-Collab-OS 产品需求文档 (PRD)

## 1. 产品概述

**产品名称：** AI-Collab-OS（AI 协作操作系统）
**版本：** v3.0（全新架构重建）
**定位：** 面向开发团队的多 AI Agent 协调开发平台

### 1.1 核心价值

用户通过配置多个 AI Agent（产品经理、架构师、前端×N、后端×N、审查员、测试、部署），
让它们像真实团队一样协作完成软件开发任务。用户是"老板"，AI 是"员工"。

### 1.2 目标用户

- 独立开发者：用 AI 团队替代人力，快速出原型
- 小团队：用 AI 补充缺失角色（没有专职架构师？拉一个 AI 架构师）
- 技术管理者：验证方案可行性，快速 POC

## 2. 功能需求

### 2.1 角色管理

| 功能 | 描述 | 优先级 |
|------|------|--------|
| 预设角色库 | 产品经理、架构师、前端、后端、审查员、测试、部署 | P0 |
| 自定义角色 | 用户可创建新角色（名称、图标、System Prompt、技能描述） | P0 |
| 角色实例化 | 同一角色可创建多个实例（前端1号~5号） | P0 |
| 角色配置 | 每个实例可独立配置 LLM Provider/Model/API Key | P0 |
| 角色删除/停用 | 项目进行中可动态增减角色 | P1 |

| 功能 | 描述 | 优先级 |
|------|------|--------|
| 动态 DAG 工作流 | 按项目需求配置执行流程图（谁先谁后、谁并行） | P0 |
| 任务分配 | 架构师可将任务拆分分配给多个前端/后端 | P0 |
| Agent 间对话 | Agent 之间可以互相发消息、讨论、反馈 | P0 |
| 审查循环 | 审查员打回 → Coder 修改 → 再审查，直到通过 | P0 |
| 多架构师讨论 | 2个架构师互相讨论出方案后再往下传 | P1 |
| 自动模式 | 全自动执行工作流，无需人工干预 | P1 |
| 手动模式 | 用户可逐步控制，手动触发下一步 | P0 |
| 人工介入 | 任何环节用户可插入意见、修改方向 | P0 |

### 2.3 通信与可见性

| 功能 | 描述 | 优先级 |
|------|------|--------|

| 功能 | 描述 | 优先级 |

## 3. 非功能需求

| 维度 | 要求 |
|------|------|
| 性能 | 单项目支持 20+ Agent 并发；消息延迟 < 500ms |
| 可扩展性 | 新角色/新 Provider 可通过配置添加，无需改代码 |
| 可靠性 | 工作流中断可恢复（检查点机制） |
| 安全性 | API Key 加密存储；JWT 认证；CORS 限制 |
| 可部署性 | Docker 一键部署；支持本地和云端 |

## 4. 用户行为流程

### 4.1 典型使用流程

```
1. 用户登录 → 创建项目 → 填写项目任务书
2. 配置团队：拉入 PM×1, 架构师×2, 前端×5, 后端×7, 审查×2, 测试×2
3. 配置工作流 DAG：
   PM → 架构师A+B讨论 → 架构师输出方案
   → 架构师分配任务给前端1-5 + 后端1-7
   → 前端/后端并行开发
   → 审查员审查 → 打回/通过
   → 测试 → 部署
4. 启动工作流（自动/手动）
5. 实时观察 Agent 间对话和产出
6. 随时介入：修改方向、追加需求、调整人员
7. 代码确认入库 → 下载
```

### 4.2 边界条件

- Agent 调用 LLM 失败：自动重试 3 次，仍失败则暂停工作流并通知用户
- 审查循环死循环：最多 3 轮，超过则升级给用户决策
- Agent 产出内容过长：截断 + 摘要
- 多 Agent 同时修改同一文件：冲突检测 + 用户手动合并

### 4.3 移动端需求

移动端为必要功能，定位为"监控+轻操作"：
- 实时查看 Agent 工作状态和对话流
- 接收通知（审查打回、需要介入、工作流完成）
- 轻量操作：审批、发消息给 Agent、启停工作流
- 查看代码产出（只读）
- 不需要在手机上写代码/编辑文件

技术方案：响应式 Web（PWA），不做原生 App。

### 4.4 代码执行环境

Agent 必须能真正运行代码，不能只生成文本：
- 每个项目分配一个 Docker 沙箱容器
- 测试 AI 可以在沙箱中执行测试脚本，获取真实测试结果
- 部署 AI 可以在沙箱中构建项目、验证部署流程
- 沙箱隔离：项目间互不影响，资源限制（CPU/内存/时间）
- 执行结果实时回传给 Agent 和用户

### 4.5 协同编辑与冲突管理

多 Agent 同时写代码时的协作机制：
- 文件级锁：Agent 开始修改文件时自动加锁，完成后释放
- 冲突检测：两个 Agent 同时修改同一文件时，系统检测并阻止
- 冲突解决策略：
  - 自动合并（不同区域的修改）
  - 升级给架构师 AI 决策（同区域冲突）
  - 最终升级给用户手动合并
- 文件状态广播：所有 Agent 实时知道谁在改什么文件
- 变更感知：Agent 收到通知"前端2号刚修改了 App.tsx"

### 4.6 非目标（明确不做）

- 不做原生移动 App（PWA 覆盖移动端需求）
- 不做多人协作（v3.0 单用户，后续版本考虑多人）
- 不做自然语言转代码的通用 AI IDE（我们是多 Agent 协调平台，不是 Cursor）

## 5. 风险点

| 风险 | 影响 | 缓解措施 |
|------|------|----------|
| LLM API 费用失控 | 20 Agent 并发，Token 消耗巨大 | Token 预算限制 + 实时统计 |
| Agent 产出质量不稳定 | 代码质量参差不齐 | 审查循环 + 人工介入 |
| 工作流复杂度爆炸 | DAG 配置困难 | 预设模板 + 可视化编辑 |
| WebSocket 连接数过多 | 20+ Agent 实时通信压力 | Redis Pub/Sub 分发 |

---

## 6. 技术约束

- 后端：Python 3.11+ / FastAPI / LangGraph
- 前端：React + TypeScript + Vite
- 数据库：PostgreSQL（生产）/ SQLite（开发）
- 缓存/消息：Redis
- LLM：LiteLLM 统一适配（OpenAI/Anthropic/Ollama/自定义）
- 部署：Docker Compose
